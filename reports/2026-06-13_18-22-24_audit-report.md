## F1 — Frontend test suite 0-running → FIXED (High → Resolved)

**Symptom.** All 21 vitest files reported **0 tests**, each erroring *"Vitest failed to find the current
suite."* The gate was dark (i18n key-lock, `messages`, `status`, component tests guarding nothing).

**Root cause (reproduced from first principles).** A known **vitest 4.1.8 bug** ([vitest-dev/vitest
#6556](https://github.com/vitest-dev/vitest/issues/6556), [#7465](https://github.com/vitest-dev/vitest/issues/7465)):
its default **`forks`** pool (and `threads`) fails to establish the runner's suite context on this
toolchain, so **any** top-level hook throws. Evidence chain — each variable isolated:

| Hypothesis tested | How | Result |
|---|---|---|
| Project test code / `test-setup.js` | throwaway test, **no setup file**, one bare `afterEach` | **Fails** → not the project |
| The project at all | **clean-room** temp project: `vitest@4.1.8 + vite@8.0.14`, trivial test | **Fails** → not the repo/config |
| My Claude-Code sandbox | re-run with sandbox disabled | **Fails** → not the sandbox |
| **Node version** (my earlier wrong guess) | portable **Node 22.22.3**, worker confirmed `process.version=v22.22.3` | **Fails on 22 too** → **not Node** |
| Duplicate vitest internals | `npm ls` + `find` | single `@vitest/runner@4.1.8` → not a dupe |
| The worker pool | `--pool=vmThreads` | **PASSES** → **the pool is the cause** |

So the variable that fixes it is the **pool**, not Node. (`vitest@3.2.6` also runs — confirming a v4-pool
regression — but it isn't a clean drop-in: `plugin-react@6` doesn't apply the automatic JSX runtime under
vitest 3's older bundled vite → 77 `React is not defined`, plus 5 high-severity advisories. Rejected.)

**The fix (applied, all deps kept).** In [`webui/vite.config.js`](../webui/vite.config.js), `test` block:
```js
pool: 'vmThreads',   // avoids the forks/threads suite-context bug
globals: true,       // lets @testing-library/react auto-register cleanup() in-test-context
```
- `vmThreads` alone → 175/216 (the 41 failures were all *"Found multiple elements"* = `cleanup()` not
  firing, because the **setup-file** `afterEach` doesn't run under the broken hook path).
- Adding `globals: true` makes `@testing-library/react` register its cleanup at **import time inside each
  test module** (that path *does* fire under vmThreads — proven by the probe), so the DOM is cleaned
  between tests. Together: **216/216 green.**
- Verified the config alone (plain `npm test`, no CLI flags) → 216 pass, **and** `npm run build` still
  builds (the `test` block doesn't touch the build). No dependency changed; no advisories.

**Why it was "green this morning, dark now":** unproven, but the repo + `node_modules` are byte-identical
since 8 Jun, so the prior `10-22-20` pass's "216 passed" was either run on a different host/Node patch or
not truly executed. Either way it's moot — the suite is now genuinely green here and the config is
host-robust. **Remove `pool`/`globals` once vitest ships a fix** (comment in the config points here).
