## What was verified clean (with evidence)

**Security**
- **No injection surface:** `grep` for `subprocess|os.system|os.popen|eval|exec|pickle|shell=True|yaml.load` across `*.py` returns only a test (`tests/test_file_lock.py:179`, list-form `subprocess.Popen`, safe) and unrelated `_pool_exec`/`_exec` method names. Confirmed.
- **No web XSS sinks:** `grep` for `dangerouslySetInnerHTML|innerHTML|eval|new Function|insertAdjacentHTML|document.write` over `webui/src` → **no matches**. React escapes all rendered node names/tags. Confirmed.
- **Content-before-extension gating:** `UniversalImporter.verify_content_matches_extension` (`universal_importer/importer.py:53`) refuses EXE/ELF/Mach-O/shebang signatures and magic/extension mismatches on **both** the path branch (`importer.py:110`) and the bytes branch (`importer.py:129`). Confirmed.
- **No path traversal on archive members:** member names are used only as tree **labels** (`archives.py:89,145`); content stays in `BytesIO`, never written to a member-named path. The bytes converter validates `ext` against the supported whitelist (`importer.py:124`) *before* using it as a `NamedTemporaryFile` suffix (`importer.py:131`), so a hostile suffix can't escape temp. No Zip-Slip. Confirmed.
- **Bomb guards intact:** zip/tar enforce member-count (500) + declared *and* actual decoded-byte caps (500 MB) with a dedicated `_ArchiveTooLarge` that the per-member catch can't swallow (`archives.py:38,77-84,134-140`); email path caps parts + a running decoded total incl. the body (`archives.py:226-283`, msg `294-334`). Confirmed.
- **Temp-name sanitization:** `materialize_subset` strips `<>:"/\|?*`, Windows-reserved basenames, and trailing dots/spaces from the tag-derived name before it becomes a path (`core/api.py:885-892`). Confirmed.
- **Fail-loud, not blank:** WebView2-runtime registry probe + native dialog/download fallback (`host.py:314-351`); Mark-of-the-Web `.NET` load failure gets a plain unblock hint (`host.py:354-404`). PII kept out of logs in the email path (`archives.py:198,219`). Confirmed.

## Rejected candidate (investigated, NOT a finding)

**Suspected:** the uncommitted `txt_to_pdf` rewrite (`universal_importer/converters.py:64`)
wraps a line character-by-character with `c.stringWidth(cur + ch, ...)`, which *looked*
like O(L²) per line (a CPU-DoS risk on a crafted single-very-long-line `.txt`).

**Why it's not a bug:** `cur` is flushed to `out` and reset to a single char the moment
adding a char would exceed `max_width`, so `cur` never grows beyond the column width
(~68 chars for Helvetica 10 on A4). Each `stringWidth` call therefore measures a
bounded string → the wrap is **O(L)**, not O(L²). The fix preserves all characters
(no truncation, no off-page clipping) without a performance regression. Verified by
reading the loop at `converters.py:64-76` and confirmed by the passing regression test
`test_txt_long_line_is_wrapped_not_truncated`.
