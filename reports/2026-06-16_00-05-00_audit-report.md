## Overall assessment (readability & structure)

This is a **mature, well-structured codebase** with a clear, correctly-applied
layering: a pure, immutable, data-driven domain (`core/model.py`, `core/commands.py`),
a port/adapter `Engine` boundary (`core/engine.py`), stateful infra isolated in
`services/`, format I/O in `formats/`, and a UI-free frontend logic layer
(`webui/src/lib/`). The reducer pattern in `core/commands.py` (frozen dataclass
commands + a `@_handler`-registered pure `apply(doc, cmd, engine)`) is **exemplary**
and is the model the rest of the project should aspire to. Comment discipline is
strong — comments explain the *why* (lock ordering, race conditions, GIL caveats),
matching the workspace style.

The findings below are **structure/readability refinements**, not defects. There are
**no Critical or High findings.** The two Medium items are both "single hub accreting
many responsibilities" — a natural outcome of organic growth that is now worth a
deliberate decomposition pass.

## Strengths worth preserving (do not "refactor away")

- **`core/commands.py`** — frozen-dataclass commands + decorator-registered pure
  handlers + a single `apply()` reducer. Textbook data-driven design; adding a command
  means adding data + one handler, never touching control flow. **Reference quality.**
- **`core/model.py`** — immutable `Node`/`Document` with structural-sharing transforms;
  the foundation that makes undo/redo and the render-cache version token (`crc32`) clean.
- **Port/adapter boundaries** — `Engine` (`core/engine.py`), `infra/tasks.py` execution
  port, injected renderer + CPU sampler in `services/render_service.py`. These are why the
  logic layer is fully headless-testable, and they are correctly applied.
- **Comment discipline** — comments justify non-obvious decisions (lock ordering, the
  `previewRef` callback-ref race in `App.jsx:70-90`, GIL caveats) rather than restating code.

## Summary

**Findings by severity:** Critical 0 · High 0 · Medium 2 · Low 2 (total 4).

**Top priority fixes (structure):**
1. **S-1** — Decompose `CoreApi` into a thin façade + `LockManager` / `ViewDirManager` /
   render-prefetch coordinator. Biggest single readability lever in the codebase.
2. **S-2** — Extract `useTagView` / `useSelection` / `useDialogs` hooks out of `App.jsx`.
3. **C-1** — Add `.eml`/`.msg` fixtures to raise `archives.py` coverage off 58 %.

**Architectural improvements worth considering:**
- The S-1 collaborators (`LockManager`, render coordinator) are not just readability wins —
  they become **independently unit-testable units**, narrowing the surface each test must
  set up (today many tests instantiate the whole `CoreApi` to exercise one concern).
- Consider whether the render/prefetch coordinator extracted in S-1 should live under
  `services/` (where the other stateful render infra already is), leaving `core/api.py`
  purely as the JSON request/response boundary — sharpening the existing `core` vs
  `services` split.

**Quick wins (high impact, low effort):**
- **D-1** — section-banner comments in `core/api.py` (minutes; no risk).
- **S-2** — the hook extractions are mechanical and fully covered by `App.*.test.jsx`.
